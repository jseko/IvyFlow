# IvyFlow Phase 2: Deep Design

## Steps

### 0. Entry State Verification

```bash
ivy state show --change "<name>"
```

Verify current phase is `design`. If not, return to the correct phase skill.

**Precondition check**: Verify that open-phase artifacts exist before proceeding:

```bash
ls openspec/changes/<name>/proposal.md openspec/changes/<name>/design.md openspec/changes/<name>/tasks.md
```

If any of these files are missing, the change was not properly initialized in the open phase. Stop and prompt the user to run `/ivyflow-open` first.

### 1. Generate Context Handoff

Generate the OpenSpec → Superpowers handoff package. Must be generated by script — agent-written summaries are not allowed.

```bash
ivy handoff "<change-name>" design --write
```

This creates:
- `openspec/changes/<name>/.ivy/handoff/design-context.json` — machine index with change, phase, file list, SHA256
- `openspec/changes/<name>/.ivy/handoff/design-context.md` — human-readable context with traceable markers

### 2. Execute Brainstorming

Before brainstorming, verify project context completeness using `references/context-six-questions.md`. If any of the six questions cannot be answered, pause and clarify before proceeding.

**Immediately execute:** Use the Skill tool to load the Superpowers `brainstorming` skill. Skipping this step is prohibited.

When loading the skill, provide context:

```
Change: <change-name>
OpenSpec Context Pack: openspec/changes/<name>/.ivy/handoff/design-context.md
Machine handoff: openspec/changes/<name>/.ivy/handoff/design-context.json
```

Follow the brainstorming skill's guidance:
- Clarifying questions to understand the problem space
- 2-3 technical approaches with trade-offs
- Step-by-step design confirmation
- Do not write the Design Doc until the user confirms the approach

If the Superpowers `brainstorming` skill is unavailable, stop and prompt to install Superpowers. Do not substitute with normal conversation.

### 3. User Confirms Design (Blocking Point — Developer Decision Required)

After brainstorming produces a design proposal, pause and wait for user confirmation:
- Technical approach adopted
- Key trade-offs and risks
- Testing strategy
- Spec patches to write back (if any)
- Spec quality: run the 10-item checklist in `references/spec-quality-check.md`

### 4. Delta Spec Completeness Checklist

After the user confirms the design, run the 5-category delta spec completeness check. This step SHALL NOT be skipped for `full` workflow.

For each category, mark as `covered`, `missing`, or `not_applicable`:

| # | Category | Check |
|---|----------|-------|
| **D1** | Input Boundaries | Are empty values, whitespace-only, overlong input, and special characters handled in the spec? |
| **D2** | External Dependency Exceptions | Are API timeout, storage unavailable, network disconnect, and quota exceeded scenarios covered? |
| **D3** | Partial Failure Degradation | When a batch operation partially fails, how are remaining items handled? |
| **D4** | Security Defaults | Does the spec enforce safe patterns (textContent over innerHTML, parameterized queries, CSRF tokens)? |
| **D5** | Multi-Path Guards | Are all input paths guarded (Enter key, keyboard shortcuts, API calls — not just the primary UI path)? |

Record results in `.ivy/handoff/delta-spec-checklist.md`:
```markdown
# Delta Spec Completeness Checklist
- Change: <change-name>
- Date: <YYYY-MM-DD>

| Category | Status | Notes |
|----------|--------|-------|
| D1 Input Boundaries | covered / missing / n/a | ... |
| D2 External Dependency Exceptions | ... | ... |
| D3 Partial Failure Degradation | ... | ... |
| D4 Security Defaults | ... | ... |
| D5 Multi-Path Guards | ... | ... |
```

For each category marked `missing`:
- Write the missing scenarios as new `#### Scenario:` entries in the corresponding `specs/*/spec.md` file
- Include the spec patch in the checklist notes
- After writing, regenerate handoff to update the hash: `ivy handoff "<change-name>" design --write`

### 5. Create Design Doc

Create the Design Doc based on the confirmed approach at `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`.

### 6. Update State

```bash
ivy state set design_doc "docs/superpowers/specs/YYYY-MM-DD-topic-design.md"
```

If delta spec files were modified during brainstorming or checklist, regenerate handoff to update hash:

```bash
ivy handoff "<change-name>" design --write
```

### 7. Guard and Transition

```bash
ivy guard design --apply
```

Guard checks: handoff_context exists, handoff_hash matches, delta-spec-checklist.md exists (for full workflow).

### 8. Resolve Next Skill

```bash
ivy next "<change-name>"
```

If `NEXT: auto, SKILL: ivy-build`, automatically load the `ivy-build` skill.
