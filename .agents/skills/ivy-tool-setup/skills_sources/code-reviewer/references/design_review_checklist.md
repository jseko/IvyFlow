# Design Review Checklist

用于评审设计文档（design.md、架构方案、API 设计等），而非代码。

## 架构决策

- [ ] 决策是否说明了 **为什么选 X 而不选 Y**（备选方案 + 排除理由）
- [ ] 决策是否与项目现有技术栈和约束一致
- [ ] 是否考虑了可扩展性（能否在不修改现有代码的情况下扩展）
- [ ] 是否考虑了可逆性（决策失败时能否回退）

## API 设计

- [ ] API 路径命名是否遵循 RESTful 规范（资源名复数、动词用 HTTP method）
- [ ] 请求/响应结构是否明确定义（DTO 字段、类型、必填/可选）
- [ ] 是否考虑了分页、筛选、排序等通用查询需求
- [ ] 错误响应是否统一格式（错误码 + 消息 + 详情）
- [ ] 是否标注了权限要求（哪些角色/权限码可访问）

## 数据模型

- [ ] 表/实体命名是否一致（项目约定：`sys_` 前缀系统表、业务表无前缀）
- [ ] 字段类型是否合理（枚举用 String/Integer、金额用 Decimal、时间用 DateTime）
- [ ] 是否考虑了索引策略（查询热点字段、联合索引）
- [ ] 关联关系是否明确（一对一、一对多、多对多，外键约束）
- [ ] 是否考虑了软删除 / 状态字段 / 审计字段（createdAt、updatedAt）

## 安全设计

- [ ] 是否标注了认证要求（JWT / Session / 公开接口）
- [ ] 是否标注了授权要求（角色 / 权限码 / 数据范围）
- [ ] 敏感数据是否考虑了加密存储和脱敏展示
- [ ] 是否考虑了输入校验（参数类型、长度、格式）
- [ ] 是否考虑了速率限制和防重放

## 性能与可靠性

- [ ] 是否识别了性能瓶颈和高频接口
- [ ] 缓存策略是否合理（缓存粒度、TTL、失效机制）
- [ ] 是否考虑了并发场景（锁、事务隔离级别、幂等性）
- [ ] 是否考虑了降级和容错（依赖不可用时的 fallback）

## 可维护性

- [ ] 设计是否遵循 SOLID 原则（单一职责、开闭原则等）
- [ ] 是否避免了过度设计（YAGNI：只实现当前需要的功能）
- [ ] 是否消除了重复设计（DRY：相似功能统一抽象）
- [ ] 变更影响范围是否可控（修改一处是否影响多处）

## 风险与迁移

- [ ] 是否列出了已知风险和缓解措施
- [ ] 是否有迁移计划（数据迁移、接口兼容、灰度策略）
- [ ] 是否有回滚方案
