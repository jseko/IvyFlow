## §4 需求分析 Agent 规范

### 能力边界声明

需求分析**不使用独立 Agent**。核心理由：需求信息来源于与用户的直接对话，独立 Agent 无法获取对话中的微妙上下文，两次传递会丢失关键细节。需求分析由主 agent 在对话上下文中直接执行，确保信息保真度。

| 能力 | 执行者 | 说明 |
|------|-------|------|
| 需求解析与提炼 | 主 agent | 从对话中提取需求要点 |
| 提案名称生成 | 主 agent | kebab-case 规则 |
| proposal.md 生成 | 主 agent | 填充 proposal-template.md |
| 会议纪要解析 | 主 agent + 知识引擎 | 结构化提取 → 检索匹配 → 生成草案 |
| 业务流程图绘制 | 主 agent | 根据 ASCII 图规范绘制 |

### 输入源 → 输出映射

| 输入源 | 处理方式 | 输出 |
|--------|---------|------|
| 用户自然语言描述 | 主 agent 解析提炼 | proposal.md Why / What Changes |
| 用户提供的外部文档 | 阅读 + 结构化提取 | proposal.md Capabilities |
| 会议纪要/需求文档 | AI 抽取器 → 知识引擎匹配 → 草案生成 | proposal.md + 业务流程图 |
| P0 技术栈变量 | 自动注入 | design.md 技术对齐章节 |

### 会议纪要解析流程

```
会议纪要/需求文档输入
    │
    ▼
[AI 抽取器] 结构化提取
    │  提取：功能点、角色、业务流程、约束条件
    │  输出：结构化需求卡片
    ▼
[知识引擎] 检索匹配
    │  查询 knowledge/business/ 已有业务知识
    │  匹配 knowledge/memory/INDEX.md 历史记忆
    │  检索阈值：≥ 2 标签匹配
    ▼
[草案生成] 组装 proposal
    │  合并抽取结果 + 知识库参考
    │  生成业务流程图（ASCII）
    │  标注不确定项供用户确认
    ▼
用户确认 → 进入步骤 1.4（design.md）
```

### 业务 ASCII 图绘制规范

> 统一规范详见 [`references/ascii-diagram-conventions.md`](ascii-diagram-conventions.md)

### §4.X 5 类高频遗漏场景强制检查表（Mandatory Gap Checklist）

> **来源**：Superpowers brainstorming skill 的反向提问模式。AI 在需求澄清阶段最高频遗漏的 5 类场景的结构化检查表。

#### 检查表定义

| # | 场景类型 | 核心问题模板 | 触发条件 |
|---|---------|------------|---------|
| G1 | **异常路径与降级** | "当 {核心操作} 失败时（超时/服务不可用/数据异常），系统应如何响应？用户侧呈现什么？" | 涉及外部依赖或异步操作 |
| G2 | **并发与幂等性** | "{操作} 是否可能被同一用户/系统重复触发？需要幂等保障吗？" | 涉及写操作、支付、状态流转 |
| G3 | **权限矩阵** | "哪些角色可以执行 {操作}？角色之间的数据隔离边界是什么？" | 涉及多角色或数据访问控制 |
| G4 | **数据约束与迁移** | "现有 {表/字段} 是否有历史数据需要迁移？新字段对旧数据的默认值是什么？" | 涉及数据库变更 |
| G5 | **接口版本与兼容性** | "该接口是否有存量调用方？变更是否需要保持向后兼容？" | 涉及接口变更或新增 |

#### 遍历规则

```
对于每类场景（G1-G5）：
    IF 当前需求命中触发条件：
        IF 用户已在需求描述中提供答案 → 标记为"已覆盖"，不重复提问
        ELSE → 转化为 AskUserQuestion 问题，加入待问列表
    
合并同类项后，最多输出 4 个 AskUserQuestion
超出 4 个时，按 G2 > G1 > G3 > G4 > G5 优先级截断
剩余未问的场景，在 proposal.md 的"待确认事项"章节列出
```

#### 优先级截断策略

G2（并发/幂等性）和 G1（异常路径/降级）是生产事故最高频来源，优先保证在 AskUserQuestion 中覆盖。G4（数据约束/迁移）和 G5（接口版本/兼容性）信息密度较高但通常可以从代码检索引申，允许写入待确认事项。

#### 与现有流程的衔接

- **探索快速通道步骤 A**：在 AskUserQuestion 交互式需求澄清前先跑检查表，命中的 Gap 合并进提问列表（总数 ≤ 4）
- **标准路径步骤一 1.3**：生成 proposal.md 前执行检查表，未提问的 Gap 写入 proposal.md §待确认事项

## §6 设计 Agent 规范

### Plan Agent 条件触发逻辑

设计阶段**默认由主 agent 执行**，仅在满足以下**任一**条件时触发 Plan Agent：

| 触发条件 | 说明 | 示例 |
|---------|------|------|
| **> 5 个 Entity** | 新增或修改的实体类超过 5 个 | 新增用户/角色/权限/菜单/日志 5+ 实体 |
| **跨模块变更** | 涉及 ≥ 3 个后端模块 或 ≥ 3 个前端页面 | 用户管理 + 权限管理 + 菜单管理 |
| **新架构模式** | 引入新的设计模式或架构组件 | 引入事件驱动、CQRS、多数据源 |
| **DB Schema 重构** | 涉及表结构变更 > 3 张表或新增外键关联 | 拆分/合并表、新增多对多关联表 |

**不触发 Plan Agent 的场景**（主 agent 直接设计）：

- 单模块 CRUD 新增（≤ 5 个实体）
- 单个 API 接口新增/修改
- 配置变更
- Bug 修复
- 前端单页面/组件新增

### GitNexus 增强（设计阶段）

> 以下接口通过 §2 代码智能层调用，CI_MODE=no 时自动 Fallback。

| 接口 | 用途 | 触发时机 |
|------|------|---------|
| `ci_clusters(pattern)` | 分析现有模块结构，辅助新模块划分 | 设计模块拆分时 |
| `ci_processes(query)` | 追踪现有业务流程调用链，识别影响范围 | 修改现有流程时 |
| `ci_query(question)` | 问答式查询现有代码约定和模式 | 技术方案不确定时 |

### 系统架构图 ASCII 规范

> 统一规范详见 [`references/ascii-diagram-conventions.md`](ascii-diagram-conventions.md)

```
┌── 层名称 ────────────────────────────┐
│  [组件A]                             │
│  [组件B]          [组件C]            │
└──────────────┬───────────────────────┘
               │ 协议/调用方式
┌──────────────▼───────────────────────┐
│  [下层组件]                          │
└──────────────────────────────────────┘
```

- 层次从上到下：前端层 → API 网关层 → 业务服务层 → 数据层
- 组件用 `[组件名]` 标记
- 连接线标注协议（HTTP/RPC/消息队列）

### ER 图 ASCII 规范

> 统一规范详见 [`references/ascii-diagram-conventions.md`](ascii-diagram-conventions.md)

```
[EntityA]                      [EntityB]
─────────────                  ─────────────
+ PK id                        + PK id
- field_1                      - FK entity_a_id ────┐
- field_2                      - field_1            │
   1 │                                               │
     │ 1..*                                          │ *..1
     ▼                                               │
[EntityC]                         ┌──────────────────┘
─────────────                     │
+ PK id                           │
- FK entity_a_id ─────────────────┤
- FK entity_b_id ─────────────────┘
```

- PK 用 `+` 前缀，FK 用 `- FK` 标注
- 关联线标注基数（1 / 0..1 / 1..* / *..*）
- FK 关系线指向关联表的 PK

## §7 测试/诊断 Agent 规范

### Spec → Test 派生规则

根据 spec.md 中的章节类型，按以下映射表派生测试用例：

| Spec 章节 | 单元测试（UT） | 集成测试（IT） | E2E 测试（ET） | 说明 |
|-----------|:---:|:---:|:---:|------|
| Scenario（业务场景） | ✅ | ✅ | ✅ | Happy Path 是 E2E 的核心输入 |
| Rule（业务规则） | ✅ | ✅ | — | 规则验证以 UT+IT 为主 |
| Boundary（边界条件） | ✅ | ✅ | ✅ | 边界值是 E2E 的重要覆盖点 |
| Exception（异常类型） | ✅ | ✅ | — | 异常路径不产生用户可见结果 |
| API 接口定义 | — | ✅ | — | 接口测试天然对应 IT |

### 失败诊断流程

```
测试失败
    │
    ▼
[步骤 1] 分类失败类型
    ├── 编译错误 → 步骤五处理
    ├── 断言失败 → 进入步骤 2
    └── 运行时异常（NPE等）→ 进入步骤 2
    │
    ▼
[步骤 2] 调用 ci_context 展开上下文
    ci_context(failed_file, depth=2)
    │
    ▼
[步骤 3] 调用 ci_processes 追踪调用链
    ci_processes("{失败方法名}")
    │
    ▼
[步骤 4] 生成诊断报告
    ┌──────────────────────────────────────┐
    │ 失败用例：UT-003                      │
    │ 失败方法：UserService.findById        │
    │ 根因：Mock 数据未设置 id=null 场景     │
    │ 影响范围：仅该用例                     │
    │ 修复建议：补充 null 参数测试用例       │
    └──────────────────────────────────────┘
```

### 回归影响分析

```
ci_impact(changed_file, "upstream")
    ├── CRITICAL → 必须运行全部关联测试
    ├── HIGH → 运行直接上游的集成测试
    └── MEDIUM/LOW → 运行变更文件的单元测试即可
```

### 能力边界声明

| 能力 | 执行者 | 说明 |
|------|-------|------|
| 测试用例生成 | 主 agent | 按 Spec→Test 派生规则生成 |
| 单元测试实现 | 主 agent 或 spring-agent | 简单测试直接生成，复杂 Mock 场景用 Agent |
| 测试失败诊断 | 主 agent + ci_context | GitNexus 可用时增强，不可用时手动分析 |
| E2E 测试执行 | 主 agent | 运行 playwright/cypress 命令，分析结果 |
| 回归影响评估 | ci_impact | 精确分析变更影响范围 |

## §5.4 Agent Prompt 精简模板（v3.2 新增）

> 为减少 Token 消耗，步骤三 Agent 调用使用以下精简模板（约 1400 tokens），替代完整版模板（约 3000 tokens）。精简模板省略重复的编码约定，改为引用项目规则文件。

### 后端任务（spring-agent）精简版

```
Task(
  subagent_name="spring-agent",
  description="创建 {类名}",
  prompt="
    项目：Spring Boot 1.5.9 + JPA + Shiro，包名 com.shimh
    文件：{FILE_PATH}
    任务：{任务描述}
    设计参考：openspec/changes/{提案名称}/design.md
    编码约定：遵循 .claude/rules/java-spring-boot.md 规范
    关键约束：不使用 Lombok | @Autowired 字段注入 | Result 成功码 0 | Entity extends BaseEntity
    请直接创建代码文件。"
)
```

### 前端任务（frontend-agent）精简版

```
Task(
  subagent_name="frontend-agent",
  description="创建 {组件名}",
  prompt="
    项目：Vue 2.5 + Element UI 2.0 + Vuex + Axios
    文件：{FILE_PATH}
    任务：{任务描述}
    设计参考：openspec/changes/{提案名称}/design.md
    编码约定：遵循 .claude/rules/vue.md 规范
    关键约束：Options API | res.code === 0 | .sync 双向绑定 | scoped 样式 | el- 前缀
    请直接创建代码文件。"
)
```

**使用规则**：
- standard / full 模式：使用精简模板（节省 ~55% token）
- quick 模式：主 agent 直写，不使用 Agent（节省 100% Agent 调用 token）
- 复杂任务（>3 个文件或有复杂业务逻辑）：可降级使用完整模板

### §5.4.X TDD 强制约束（仅 ENABLE_UNIT_TEST=on 时注入）

> 当 `{ENABLE_UNIT_TEST}=on` 时，在 §5.4 精简模板的"关键约束"字段中追加以下内容。

#### Iron Law

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

#### 红绿重构循环

```
RED   → 先写测试用例 → 运行确认失败（非编译错误）
GREEN → 实现最小代码使测试通过 → 运行确认通过
REFACTOR → 消除重复、优化命名（不改变行为）→ 重跑确认仍绿
```

#### 任务粒度分级

| 任务类型 | TDD 要求 |
|---------|---------|
| Service / Domain（业务逻辑） | 完整红绿重构循环 |
| Repository / Mapper（数据访问） | 红绿两步（重构可选） |
| Controller / API 层 | 红绿两步，优先集成测试 |
| 纯 DTO / Entity（< 50 行） | 豁免，间接覆盖 |
| 配置文件 / Migration SQL | 豁免 |

#### Agent Prompt 追加内容

在原有"关键约束"行后追加：

```
TDD 强制要求（必须遵守）:
- Iron Law: 没有失败测试前，不写一行实现代码
- 红绿重构循环: RED→GREEN→REFACTOR
- 任务粒度: {根据上表判断当前任务类型}
- 禁止: 先写实现后补测试 | "just this once" 跳过 | 声称"稍后补测试"
```

#### Rationalization 反驳表（主 Agent 步骤三 3.2 GREEN 阶段检测参考）

| 借口 | 事实 |
|------|------|
| "这个太简单了，不需要测试" | 简单代码也会坏。写测试只要 30 秒 |
| "我先实现再补测试" | 后补的测试立即通过，证明不了任何东西 |
| "后补测试能达到同样目的" | 后补测试问"这段代码做什么"，先写测试问"这段代码应该做什么" |
| "我已经手动测试过了" | 临时测试 ≠ 系统测试。无记录、无法重跑 |
| "删掉 X 小时的工作太浪费了" | 沉没成本谬误。保留不可信代码 = 技术债务 |
| "测试难写 = 设计有问题" | 听测试的。难测 = 难用。简化接口 |
| "TDD 会拖慢我" | TDD 比 Debug 快。务实的做法就是测试先行 |

#### Red Flags 检测清单（主 Agent 步骤三 3.2 GREEN 阶段执行时检查）

检测到以下任一模式 → STOP，任务标记为 TDD 违规，回到 RED：

- 实现代码先于测试代码被创建
- 测试在实现之后编写
- 测试首次运行即通过（说明覆盖的是已有行为，非新功能）
- 无法解释测试为什么失败
- 声称"稍后补测试"
- "just this once" / "这个场景例外因为……"
- "我已经手工测过了"
- "后补测试能达到同样目的"

#### Testing Anti-Patterns

| Anti-Pattern | 正确做法 |
|-------------|---------|
| 测试 Mock 行为而非真实行为 | 优先使用真实实现，Mock 仅限外部边界（网络/文件系统/时钟） |
| 为生产类添加 test-only 方法 | 通过公开接口测试；test-only 方法是设计坏味道 |
| 不理解的 Mock（为通过测试而 Mock） | 理解依赖的真实行为后再决定是否需要 Mock |
| 测试实现细节而非行为 | 测试"做了什么"（返回值/副作用）而非"怎么做的"（内部调用顺序） |
