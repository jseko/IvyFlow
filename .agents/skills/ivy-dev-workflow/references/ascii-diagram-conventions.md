# ASCII 图绘制规范

> v3.2 新增：统一所有流程/架构/ER 图的 ASCII 绘制标准。

---

## 框线字符集

### 业务流程 / 系统架构图

```
主连接线  →（单线箭头）、─（横线）、│（竖线）、┬（分支）、└（终止）

分支节点  ──→ <判断条件？>
             │是            │否
             ▼              ▼
        [处理流程]      [替代流程]

并发节点  ┌── 开始 ──┐
          │          │
        [A任务]   [B任务]
          │          │
          └── 汇合 ──┘

终止节点  [终止] 方括号，或带文字说明的终止标记
```

### ER 图 / 数据模型图

```
实体标记  [EntityName]
          ─────────────
          + PK_FIELD
          - field_1
          - field_2

关联线    1 ──── * （一对多）
          * ──── * （多对多）
          1 ──── 1 （一对一）

PK/FK    PK ──── FK （主外键关联）
          标注 FK 关系线中部
```

### 状态机图

```
状态节点  [状态名称]
转换线    ──→ 或 ──事件→
条件      ──[条件]→
开始      ●
结束      ◎
```

## 节点命名规则

| 图类型 | 节点格式 | 示例 |
|--------|---------|------|
| 流程步骤 | `[步骤名称]` | `[步骤一] 生成文档` |
| 判断节点 | `<条件描述？>` | `<文件存在？>` |
| 系统组件 | `[组件名]` | `[MySQL]` `[Redis]` |
| 用户/角色 | `角色名 →` 或 `角色名 ──→` | `用户 ──→` |
| Agent/MCP | `Agent/工具名` | `spring-agent` |
| 状态 | `{状态}` | `{ENABLE_CODE_REVIEW}` |

## 布局规范

### 通用原则

1. **层次优先**：从上到下表示时间/执行顺序，从左到右表示数据流向
2. **宽度约束**：单行不超过 80 字符（适配终端和 Markdown 渲染）
3. **对齐**：同层级节点水平对齐，父子节点垂直对齐
4. **间距**：节点间至少 1 个空格的连接线空间

### 异常流处理

- 异常流统一在正常流下方，用 `│（备选/异常）` 引出
- 异常终止节点用虚线框 `┊┊┊` 或 `···` 标记
- 异常返回主线用回环箭头示意

### 复杂度控制

- 单个图不超过 7 个层级深度
- 超过 10 个节点 → 拆分子图
- 每个判断节点不超过 3 个分支

## 示例

### 业务流程 Happy Path（规范示例）

```
用户 ──→ [登录页面] ──→ <认证成功？>
                           │是              │否
                           ▼                ▼
                     [跳转首页]        [提示错误]
                                            │
                                     <超过3次？>
                                      │是      │否
                                      ▼        ▼
                                 [锁定账户]  [返回登录]
```

### 系统架构图（规范示例）

```
┌── 前端层 ────────────────────────────┐
│  [Vue 3 App]                         │
│  [Vue Router]    [Pinia Store]       │
└──────────────┬───────────────────────┘
               │ HTTP / Axios
┌──────────────▼───────────────────────┐
│  [Spring Boot API]                   │
│  [Shiro Auth]    [Global Exception]  │
└──────┬──────────────────┬────────────┘
       │ JPA             │ Redis
┌──────▼──────┐    ┌──────▼──────┐
│  [MySQL]    │    │  [Redis]    │
└─────────────┘    └─────────────┘
```

### ER 图（规范示例）

```
[User]                      [Article]
─────────────              ─────────────
+ PK id                    + PK id
- username                 - FK author_id ────┐
- email                    - title            │
                           - content          │
   1 │                                       │
     │                                       │
     │ 1..*                                   │ *..1
     ▼                                       │
[Comment]              ┌──────────────────────┘
─────────────          │
+ PK id                │
- FK user_id ──────────┤
- FK article_id ───────┘
- content
```

## 工具参考

- 推荐使用手写 ASCII 图（无需工具依赖，MD 原生渲染）
- 复杂架构图建议使用 Mermaid 或 PlantUML，简单流程用 ASCII
- 判断节点标记 `<...>` 是 ivy-dev-workflow 专属约定，确保全局一致
