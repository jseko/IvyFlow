## §8 安全护栏

### 三层安全架构

```
┌─────────────────────────────────────────────────────────┐
│ 第一层：事前防御（P0 阶段，始终执行）                       │
│   ├── 敏感数据检测：身份证/手机号/银行卡/邮箱 PII 扫描      │
│   │   规则：security/sensitive-data-rules.md              │
│   │   命中 → 提示脱敏 or 拒绝执行                          │
│   └── Prompt 注入/越狱检测                                │
│       规则：security/prompt-injection-patterns.md          │
│       命中 → 拒绝执行 + 记录告警                           │
├─────────────────────────────────────────────────────────┤
│ 第二层：事中控制（步骤三代码实现阶段）                       │
│   ├── API 白名单核验（security/api-whitelist.yaml 存在时） │
│   ├── 组件白名单核验（security/component-whitelist.yaml）  │
│   └── Agent Prompt 嵌入安全约束前缀                         │
├─────────────────────────────────────────────────────────┤
│ 第三层：事后审核（步骤四，按执行模式触发）                    │
│   ├── 代码审查：java-reviewer / typescript-reviewer        │
│   │   触发：standard / full 模式                            │
│   ├── 安全专项审查：security-reviewer                      │
│   │   触发：仅 full 模式（ENABLE_SECURITY_REVIEW=on）       │
│   ├── 三方库 CVE 检测：基于 AI 知识（始终）                 │
│   │   ⚠️ 非实时 CVE 数据库，不可替代专业 SCA 工具           │
│   └── 审计范围界定（CI_MODE=full）                         │
│       ci_detect_changes() 精确到变更文件级                  │
└─────────────────────────────────────────────────────────┘
```

### 安全 Agent 分派表

| 阶段 | Agent | 触发条件 | 检测内容 |
|------|-------|---------|---------|
| 事前 | 主 agent | 始终执行 | PII 检测、Prompt 注入检测 |
| 事前 | 主 agent | full 模式 + 白名单文件存在 | API/组件白名单核验 |
| 事后 | `java-reviewer` / `typescript-reviewer` | standard / full | 代码质量、安全漏洞 |
| 事后 | `security-reviewer` | full 模式 | 安全专项深度审查 |
| 事后 | 主 agent | 始终执行 | 三方库 CVE 检测（AI 知识） |

### AI 扫描局限声明

> ⚠️ **重要免责声明**：步骤四中的"静态漏洞扫描"由 AI 模拟执行，**不等同于专业 SAST 工具**（如 SonarQube、Checkmarx、Fortify）。存在以下局限：
> - **误报风险**：AI 可能将安全代码标记为漏洞
> - **漏报风险**：AI 无法覆盖所有 CWE/OWASP Top 10 漏洞类型
> - **无运行时分析**：无法检测运行时注入、内存泄漏等动态问题
> - **非实时 CVE 数据**：依赖训练数据截止日期的漏洞知识
>
> **生产部署前必须使用专业安全审计工具复核。** AI 安全扫描仅作为开发阶段的辅助检查手段，**不可替代专业安全审计**。

## §9 可观测性与成本管控

### 追踪指标

| 指标 | 说明 | 采集方式 |
|------|------|---------|
| 步骤耗时 | 每个步骤的实际执行时间 | 步骤开始/结束时记录时间戳 |
| Token 消耗 | 当前步骤的 Token 消耗量 | Agent 任务返回的 `total_tokens` |
| Agent 调用次数 | 每个 Agent 类型的调用次数 | 步骤三/四的 Task 调用计数 |
| 并行效率 | 并行组数 vs 串行化任务数 | 步骤三的并行窗口统计 |
| Fallback 触发次数 | GitNexus/Agent 降级次数 | ci_impact Fallback、Agent fallback 日志 |
| 用户交互次数 | 确认/歧义/暂停次数 | Brainstorming 选择、步骤二/七/九 + 3.4 歧义询问 |

### 步骤级 Token 细分（v3.2 新增）

> 每步骤的预估 Token 消耗，含上下文累积系数 1.4x。

| 步骤 | quick | standard | full | 主要消耗来源 |
|------|------:|------:|------:|------------|
| P0 技术栈检测 | 500 | 500 | 500 | 配置文件扫描 + 变量展示 |
| 步骤一（文档生成） | 8,000 | 8,000 | 8,000 | proposal + design + specs + tasks |
| 步骤二（用户确认） | 200 | 200 | 200 | 交互模板输出 |
| 步骤三（代码实现） | 8,000 | 12,000 | 16,000 | Agent 调用（quick 主 agent 直写为主） |
| 步骤四（代码审查） | — | — | — | — |
| 步骤五（编译验证） | 500 | 500 | 500 | 编译命令执行 |
| 步骤六（测试执行） | — | — | — | — |
| 步骤七（功能确认） | 200 | 200 | 200 | 交互模板输出 |
| 步骤八（实现报告） | 1,000 | 1,000 | 1,000 | 报告生成 |
| 步骤九（归档） | 500 | 500 | 500 | 归档检查 + 执行 |
| **累计（含 1.4x 系数）** | **~26,000** | **~71,000** | **~89,000** | — |

> 上下文累积系数 1.4x 已计入上表。standard/full 模式增加约 7,000 tokens（TDD 红绿重构 + Delta Spec + 验证关卡）。实际消耗因代码库规模和 Agent 调用次数而异。

### Token 预算表（按执行模式）

| 执行模式 | 预估 Token | 告警阈值（1.2x） | 硬上限（1.5x） | 说明 |
|---------|-----------|-----------------|---------------|------|
| quick | ~26,000 | 31,000 | 39,000 | 仅步骤三（主 agent 直写为主） |
| standard | ~71,000 | 85,000 | 107,000 | 含 TDD + 代码审查 + 单元测试 + Delta Spec |
| full | ~89,000 | 107,000 | 134,000 | 含安全审查 + E2E + Plan Agent |

### 告警阈值

| 告警级别 | 触发条件 | 行为 |
|---------|---------|------|
| 🟡 预警 | Token 消耗达到预算的 1.2x | 输出警告，继续执行 |
| 🟠 超标 | Token 消耗达到预算的 1.5x | 暂停并提示用户是否继续 |
| 🔴 严重超标 | 单步骤 Token 超过预算的 2x | 自动降级：跳过后续可选步骤，简化输出 |

### 成本优化策略

| 策略 | 方法 | 预计节省 |
|------|------|---------|
| 上下文裁剪 | Agent Task 只注入相关设计章节，不注入完整 design.md | 20-30% |
| 阶段性摘要 | 步骤一/二输出压缩为 <500 tokens 摘要传递给后续步骤 | 15-20% |
| GitNexus 优先 | `ci_impact()` ~200 tokens vs AI 分析 ~3000-5000 tokens | ~95%（影响分析场景） |
| 简单文件直写 | Entity/DTO <50 行不送 Agent，主 agent 直接 Write | ~1400 tokens/次 |

## §10 知识记忆引擎

### 知识库分层结构

```
knowledge/
├── README.md                     # 知识库使用说明
├── business/                     # 业务领域知识
│   └── .gitkeep                  # 业务术语、流程规则、领域模型
├── dev/                          # 开发技术知识
│   └── naming-conventions.md     # 命名规范（类名/方法名/变量名/表名/API）
├── topology/                     # 系统拓扑知识
│   └── .gitkeep                  # 服务拓扑、模块依赖、接口契约
└── memory/                       # 记忆引擎
    ├── INDEX.md                  # 记忆索引（检索入口）
    └── archive/                  # 过期归档
        └── .gitkeep              # 超过 6 个月的已归档记忆
```

### 检索-去重-过期流程

```
┌──────────────────────────────────────────────────────┐
│ 检索触发                                              │
│   条件：用户需求中出现技术标签且匹配 ≥ 2 个            │
│   入口：knowledge/memory/INDEX.md                      │
│   匹配规则：标签交集 ≥ 2 → 关联记忆候选               │
│                                                      │
│ <Delta Spec 检索>                                     │
│   每次新迭代开始时:                                    │
│     ci_query 检索上次迭代的 specs/delta.md             │
│     将历史决策注入新 proposal 的 Context 中             │
│     检索范围：openspec/changes/*/specs/delta.md        │
│                                                      │
│ 去重归并                                              │
│   同一主题多条记忆 → 按日期保留最新 3 条               │
│   内容相似度 > 80% → 合并为 1 条（保留最新日期）       │
│   冲突标签 → 标记为"待确认"                            │
│                                                      │
│ 过期归档                                              │
│   最后访问距今 > 6 个月 → 移至 archive/                │
│   归档前压缩：摘要 + 标签保留，原文删除                 │
│   archive/ 中的记忆不参与检索                          │
└──────────────────────────────────────────────────────┘
```

### 记忆格式规范

每条记忆记录在 `INDEX.md` 中以表格行形式存储：

| 日期 | 提案名称 | 摘要（≤50字） | 技术标签 | 状态 |
|------|---------|--------------|---------|------|
| 2026-05-29 | add-user-auth | JWT + Shiro 认证方案 | auth, jwt, shiro | active |
