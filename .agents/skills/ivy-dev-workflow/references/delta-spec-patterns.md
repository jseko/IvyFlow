# Delta Spec 双层检查表与反写模板

> 本参考文档供步骤三实现阶段和步骤七汇总生成 delta.md 时使用。
> 定义"规格外决策"（第一层）和"代码级遗漏检查"（第二层）的完整检测维度和反写格式。

---

## 第一层：规格外决策（架构/业务级）

> 实现过程中有意为之的、与原始设计方案的偏离。

### 检测触发规则

| 维度 | 触发条件 | 检测方式 |
|------|---------|---------|
| **接口参数调整** | 实现时新增/修改/删除了 design.md 中未定义的接口字段 | 对比 Agent 生成的代码签名 vs design.md §接口设计 |
| **错误码新增** | 实现时新增了原始 spec 未定义的错误码或异常类型 | 对比 Agent 生成的错误类/枚举 vs spec.md |
| **实现路径变更** | 技术选型与 design.md 的技术决策偏离 | 对比 Agent 实现方式 vs design.md §Decisions |
| **第三方库选型** | 引入 design.md 未规划的依赖 | 对比 package.json/pom.xml 变更 vs design.md §技术栈 |

### 反写模板

```markdown
## 变更汇总

| # | 原规格位置 | 变更类型 | 原方案 | 实际实现 | 变更原因 |
|---|-----------|---------|-------|---------|---------|
| 1 | design.md §5.2 | 接口参数变更 | `createOrder(userId, items)` | `createOrder(userId, items, idempotencyKey)` | 支付接口需幂等保障，前端可能重复提交 |
| 2 | specs/create-order/spec.md | 错误码新增 | 未定义库存相关错误码 | `E4023: INVENTORY_RESERVED` | 库存预占失败场景未在原规格中定义 |
| 3 | tasks.md #7 | 实现路径变更 | 数据库乐观锁（version 字段） | Redisson 分布式锁 | 压测发现乐观锁在 500 QPS 下冲突率 > 30% |
| 4 | design.md §技术栈 | 第三方库选型 | 自实现哈希分片 | 引入 `xxhash` (MIT License) | 非功能需求，性能优化 |
```

## 第二层：代码级遗漏检查（实现质量级）

> 实现过程中无意为之的、对边界条件和异常路径的遗漏。

### 5 个检查维度

#### D1. 输入边界

| 检查内容 | 示例 |
|---------|------|
| 只检查 null，未处理 whitespace-only 字符串 | `if (name == null)` 未检查 `name.trim() === ""` |
| 只检查空集合，未处理 null 集合 | `if (list.isEmpty())` 未先 `if (list == null)` |
| 数值范围检查不完整 | 只检查 `> 0`，未检查 MAX_VALUE 溢出 |
| 字符串长度无上限 | 未限制用户输入最大长度 |

#### D2. 外部依赖异常

| 检查内容 | 示例 |
|---------|------|
| 存储不可用时无降级策略 | localStorage 在隐私模式下抛 QuotaExceededError |
| 网络请求无重试/超时机制 | fetch 不设置 AbortController timeout |
| 鉴权失败时无优雅处理 | JWT 过期时直接崩溃而非重定向登录 |
| 第三方 API 返回非预期格式 | 假设 API 总返回 `{data: [...]}` 但不检查字段存在性 |

#### D3. 部分失败降级

| 检查内容 | 示例 |
|---------|------|
| 批量操作采用"全部回滚"而非"跳过失败项" | 数据损坏时清空全部缓存而非跳过损坏条目 |
| 批量操作中无失败项收集和上报 | 无 skippedItems 日志/指标 |
| 部分成功时无用户通知 | 批量导入 100 条，2 条失败，只显示"导入完成" |

#### D4. 安全默认值

| 检查内容 | 示例 |
|---------|------|
| 用户输入渲染到 DOM 使用不安全 API | `innerHTML` 替代 `textContent` |
| SQL 查询使用字符串拼接 | 非参数化查询 |
| 敏感信息未脱敏 | 日志中输出完整手机号/身份证号 |
| 默认权限为"全部允许"而非"最小权限" | 新角色默认拥有所有 API 访问权 |

#### D5. 多路径守卫

| 检查内容 | 示例 |
|---------|------|
| 仅校验主交互路径（如按钮点击），遗漏等效次路径 | Enter 键提交未做空输入守卫 |
| 前端校验后，后端未做相同校验 | 前端限制了文件大小但后端未限制 |
| 一处调用加了权限检查，另一处遗漏 | API A 调 Service 时检查权限，API B 调同一 Service 时跳过 |

### 代码级遗漏反写模板

```markdown
## 代码级遗漏与加固建议

| # | 文件:行号 | 遗漏维度 | 问题描述 | 建议加固 | 优先级 |
|---|---------|---------|---------|---------|:---:|
| 1 | OrderService.java:45 | D1 输入边界 | `createOrder` 未验证 items 为空列表的情况 | 添加 `if (items == null || items.isEmpty())` 前置校验 | HIGH |
| 2 | PaymentGateway.java:78 | D2 外部依赖 | 支付接口无超时处理 | 添加 CompletableFuture.orTimeout(5, SECONDS) | CRITICAL |
| 3 | BatchImportService.java:120 | D3 部分失败 | 批量导入全部回滚，无跳过失败项选项 | 添加 try-catch per item + skippedItems 收集 | MEDIUM |
```

## 与执行模式的联动

| 模式 | 第一层（规格外决策） | 第二层（代码级遗漏） |
|------|-------------------|-------------------|
| quick | 不执行 | 不执行 |
| standard | 步骤三 3.6 记录 → 步骤七汇总生成 delta.md | 步骤七汇总时执行一次全量检查 |
| full | 每次 3.5 校验后增量追加 → 步骤七汇总 | 每次 3.5 校验后执行增量检查 |

---

## 与 {ENABLE_CODE_REVIEW} 开关的联动

- `{ENABLE_CODE_REVIEW}=on`：正常执行双层反写
- `{ENABLE_CODE_REVIEW}=off`：跳过第二层代码级遗漏检查（但输出提示："代码审查关闭，跳过代码级遗漏检查"）
- 第一层规格外决策不受此开关影响（与审查无关，是设计保真度问题）
