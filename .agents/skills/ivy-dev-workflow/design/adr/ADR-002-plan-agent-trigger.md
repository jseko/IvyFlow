# ADR-002: Plan Agent 条件触发

**状态：** 已采纳
**日期：** 2026-05-29
**决策者：** AI (评审优化)

## 上下文

v3.2 需要在设计阶段引入 Plan Agent 来增强大型项目的设计能力，但并非所有项目都需要 Agent 级别的设计支持。

三个选项：全部主 Agent 执行（简单但可能不够深入）、全部 Plan Agent 执行（成本高）、混合模式（按条件触发）。

## 决策

**混合模式——Plan Agent 条件触发。**

默认由主 Agent 执行设计，仅在满足以下任一条件时触发 Plan Agent：
- 新增/修改 > 5 个 Entity
- 跨 ≥ 3 个模块变更
- 引入新的架构模式
- DB Schema 重构（> 3 张表变更）

## 选项对比

| 选项 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 全部主 Agent | 简单，Token 成本低 | 大型项目设计可能不够完整 | ❌ |
| 全部 Plan Agent | 设计质量高，展开完整 | 小型项目过度设计，Token 成本高 | ❌ |
| 混合（条件触发） | 按需使用，成本与质量平衡 | 需要明确的触发规则 | ✅ 采纳 |

## 后果

**正面**：
- 小/中型项目不会产生不必要的 Agent 调用成本
- 大型项目获得 Plan Agent 的深度设计支持
- 4 条触发规则明确可操作

**负面**：
- 触发条件判断增加主 Agent 的决策负担
- 边界情况（如恰好 5 个 Entity）可能需要人工判断

**注意事项**：
- 触发条件在 §6 章节中明确定义
- ENABLE_PLAN_AGENT 子开关由 full 模式自动推导（conditional）
