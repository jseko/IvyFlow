# ADR-006: Token 按需引用策略

**状态：** 已采纳
**日期：** 2026-05-29
**决策者：** AI (评审优化)

## 上下文

在 agentic 工作流中，每个 Agent 调用都会消耗 Token，且上下文在多次调用中累积。需要决定是将完整的 design.md 和 specs/ 注入每个 Agent（全量注入），还是只注入相关章节（按需引用）。

全量注入的 Token 代价在 agentic 工作流中是"平方级"的——每个 Agent 都加载完整文档，随着 Agent 数量增加总成本快速增长。

## 决策

**按需引用 + 上下文裁剪。**

Agent Task 只注入与该 Agent 直接相关的设计章节：
- Entity/DTO Agent → design.md §4（数据模型）
- Controller Agent → design.md §5（API 设计）
- Service Agent → design.md §4 + §7（数据模型 + 关键流程）
- 前端 Agent → design.md §6（前端设计）

## 选项对比

| 选项 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 全量注入 | 信息完整，Agent 可见全局 | Token 代价平方级增长，上下文膨胀 | ❌ |
| 按需引用 + 裁剪 | Token 节省 20-30%，上下文精简 | 需精确划分每个 Agent 需要的章节 | ✅ 采纳 |

## 后果

**正面**：
- 预计节省 20-30% Token 消耗
- Agent 上下文更聚焦，减少无关信息干扰
- §9 可观测性章节支持按步骤追踪 Token 消耗

**负面**：
- 需要维护每个 Agent 的"相关章节"映射表
- 偶尔可能遗漏 Agent 需要的关键信息

**注意事项**：
- 4 项成本优化策略在 §9 中完整列出
- Token 预算设置了 3 级告警（1.2x 预警 / 1.5x 超标 / 2x 严重超标）
