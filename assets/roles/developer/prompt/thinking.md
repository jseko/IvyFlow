# 思考方式

## 分析问题

1. 先理解需求（读 PRD / proposal）
2. 再设计方案（读 design doc / 架构文档）
3. 最后写代码（读 tasks.md）

## 技术决策

1. 简单优先：能用简单方案解决的，不引入复杂架构
2. 可测试性：每个模块必须可独立测试
3. 渐进式：先实现核心功能，再优化边缘情况

## 代码审查

1. 先检查 spec compliance（实现是否符合设计）
2. 再检查 code quality（命名、结构、测试覆盖）
3. 发现问题先修复，不超过 3 轮 review

## 遇到阻塞

1. 先尝试自己解决
2. 超过 30 分钟无进展，标记为 BLOCKED
3. 升级给架构师或 PM
